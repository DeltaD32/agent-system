LLM Routing & Agent Prompt System

Section 3 of 4 — How tasks get routed and how agents stay in their stacks

LLM Routing Decision Tree

Task arrives at agent →
① Force override set?
user pinned backend in Hire panel
→ use that backend, done
② Is local Ollama healthy?
ping localhost GPU
→ no: try remote Ollama(s) in order
→ all down: fall back to Claude API
③ Task complexity score
keyword + token-length heuristic
LOW → local Ollama (fast, private)
MED → best available Ollama
HIGH → Claude API (reasoning tasks)
④ Role-based preference
some roles default to Claude
PM always Claude (planning)
Archivist always local (bulk ops)
Researcher local first, Claude for synthesis

Agent Prompt Anatomy — "Staying in Their Stack"

LAYER 1 — IDENTITY (never changes)
You are {role} in an AI agent office.
Your specialization: {specialization}
You ONLY perform tasks within your role.
You NEVER impersonate another agent.
If asked to do something outside your role,
you redirect to the correct specialist.
LAYER 2 — INSTRUCTIONS (from vault)
instructions/{role}.md injected here.
Contains: tools allowed, output format,
escalation rules, quality standards,
which agents to consult for what.
LAYER 3 — MEMORY (loaded per task)
Context from agents/{role}/context.md
Relevant hits from shared/ search
Current project state from PM
Previous task output (if chained)
LAYER 4 — TASK (per invocation)
The actual task description.
Expected output format.
Deadline / priority.
Which agent requested this.

Guardrail Rules (in every instructions/{role}.md)

✓ List of tools this role CAN use
✗ Explicit list of tools this role CANNOT use
✓ Output must always include: confidence score, sources, next recommended agent
✓ If blocked: write blocker to shared/ and notify PM
✗ Never write to another agent's private folder
✗ Never skip the memory-check step

Agent-to-Agent Consultation Flow (triggers pathfinding in office)

Coder-1 hits a blocker →
writes question to shared/blockers.md
emits consult_request event
sprite walks to Reviewer's desk
waits for consult_response event
resumes with answer injected into context
PM decomposes a project →
spawns N agents with specializations
emits agent_hired events (desks appear)
assigns tasks via task_queue
monitors via shared/project-status.md
dismisses agents when done (agent_dismissed)
